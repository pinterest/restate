## New Features

### New `sys_vqueue_entry_status` Table

A new `sys_vqueue_entry_status` SQL table exposes the authoritative vqueue entry status-header fields directly from the partition store. Operators can now inspect per-entry status including the entry id, vqueue id, stage, status, scheduling key fields, entry and wait statistics, deployment, memory, and retry metadata.

### New `configs` Table

A new `configs` SQL table exposes the effective runtime configuration of every node as key/value rows. The query fans out across the cluster, returning one row per configuration key on each node with the columns `plain_node_id`, `gen_node_id`, `key`, and `value`. This makes it easy to confirm what configuration a node is actually running with, and to spot drift between nodes. This is an additional mechanism alongside the existing [signal-based mechanism](https://docs.restate.dev/server/configuration#configuration-introspection).

```sql
-- Inspect a node's effective configuration
SELECT plain_node_id, key, value FROM configs ORDER BY plain_node_id, key;
```

### Optional Rate Limiting for Storage Query Execution

Datafusion query execution behind the admin REST `/query` endpoint can now be optionally rate limited via a new token-bucket `rate-limiting` knob on the query engine options. Each query consumes one token, and queries that exceed the limit fail fast with `429 Too Many Requests` and a `Retry-After` header. This lets operators cap expensive or high-volume ad-hoc SQL so it cannot starve the node's core workload of CPU and memory. Rate limiting is disabled by default, so behavior is unchanged unless explicitly configured:

```toml
[admin.query-engine.rate-limiting]
rate = "100/s"      # <rate>/<unit>, unit is s|sec|second, m|min|minute, or h|hr|hour
capacity = 200      # optional burst capacity; defaults to the rate value
```

## Improvements

### Behavioral Changes

#### Client-Side Query Errors Now Return `400 Bad Request`

The admin REST `/query` endpoint now returns `400 Bad Request` (instead of `500 Internal Server Error`) for client-side query errors such as invalid SQL or unknown columns/tables. Internal and execution failures continue to return `500`.

#### Partition Processor Command Batches Bounded by Byte Size

A new `worker.max-command-batch-bytes` option (default 1 MiB) bounds the in-memory size of a partition-processor command batch. The processor now seals a batch when it reaches either the record count (`max-command-batch-size`) or this byte size, whichever comes first. Previously batches were bounded by record count only, so deployments with large records will now form smaller batches by default, keeping the batch size more predictable. To change the cap:

```toml
[worker]
max-command-batch-bytes = "1 MiB"
```

### Stability

#### Graceful Ingress Connection Draining on Shutdown

Ingress shutdown is now a graceful drain instead of an abrupt close. On shutdown, ingress stops accepting new sockets, then asks active connections to close gracefully, giving in-flight requests up to five seconds to finish. This reduces spurious client errors during rolling restarts and upgrades.

## Bug Fixes

### HTTP/2 Stream Closure Behind Reverse Proxies

Restate's service-client connection pool previously closed HTTP/2 streams by sending an empty trailers frame. Some reverse proxies (e.g. nginx) do not handle empty trailers correctly, which surfaced as `frame with invalid size` errors for deployments sitting behind such a proxy. Streams are now closed by sending an empty DATA frame with the `END_STREAM` flag set instead, which has the same effect and is handled correctly by these proxies.
